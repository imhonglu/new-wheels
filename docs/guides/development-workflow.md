# 개발 작업 원칙

- Status: Active
- Last verified: 2026-07-21
- Verified against: `AGENTS.md`, `CONTRIBUTING.md`

이 문서는 저장소를 변경할 때 적용하는 공통 작업 원칙을 설명합니다. 사람과 코딩 에이전트 모두 같은 기준을 따릅니다.

## 구현 전에 생각하기

가정하거나 혼란을 숨긴 채 구현하지 않습니다.

- 변경 목적과 성공 기준을 먼저 정의합니다.
- 확실하지 않은 전제는 명시하고 확인합니다.
- 여러 해석이 가능하고 결과가 달라진다면 선택지를 드러냅니다.
- 더 단순한 접근이 있다면 먼저 검토합니다.
- 작업 범위를 크게 바꾸는 불명확성이 있으면 구현 전에 질문합니다.

## 단순함 우선

문제를 해결하는 최소한의 코드와 문서만 작성합니다.

- 요청되지 않은 기능을 추가하지 않습니다.
- 한 번만 쓰이는 코드에 추상화를 도입하지 않습니다.
- 필요가 확인되지 않은 유연성이나 설정을 추가하지 않습니다.
- 실제로 발생할 수 없는 상황을 위한 방어 코드를 만들지 않습니다.
- 더 짧고 직접적인 구현이 가능하면 단순한 쪽을 선택합니다.

## 최소 범위 수정

모든 수정은 작업 목적과 직접 연결되어야 합니다.

- 관련 없는 코드, 주석, 포맷을 함께 정리하지 않습니다.
- 기존 스타일을 존중하고 요청되지 않은 리팩터링을 피합니다.
- 기존 사용자 변경을 덮어쓰거나 되돌리지 않습니다.
- 현재 변경으로 사용되지 않게 된 코드만 함께 제거합니다.
- 관련 없는 문제를 발견하면 수정하지 않고 별도로 알립니다.

## 목표 중심 실행

작업을 검증 가능한 결과로 표현합니다.

- 검증 로직 추가 → 잘못된 입력을 재현하는 테스트를 작성하고 통과시킵니다.
- 버그 수정 → 실패를 재현한 뒤 변경 후 통과하는지 확인합니다.
- 리팩터링 → 변경 전후에 관련 테스트와 타입 검사가 통과하는지 확인합니다.

여러 단계가 필요하면 각 단계에 확인 방법을 붙입니다.

```text
1. 작업 → 검증: 확인 방법
2. 작업 → 검증: 확인 방법
3. 작업 → 검증: 확인 방법
```

여러 세션이나 PR에 걸치는 작업은 [실행 계획](../plans/README.md)에 기록하고 진행 상태를 갱신합니다. 작업 중 장기적으로 유지할 구조적 선택이나 대안과 트레이드오프가 생기면 계획에만 남기지 않고 [ADR](../decisions/README.md)로 분리합니다.

## 완료 조건

- 요청한 결과가 구현되었는가?
- 변경 범위가 성공 기준을 넘어가지 않았는가?
- 변경 유형에 맞는 검증을 실행했는가?
- 실행하지 못한 검증이나 남은 위험을 명확히 알렸는가?
- 저장소에 남긴 Plan이 있다면 상태와 최종 검증 결과를 갱신했는가?
- 장기적으로 유지할 결정이 있다면 ADR로 기록했는가?
